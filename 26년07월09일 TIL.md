1. 함수 오버로드: 
    컴파일러는 함수 이름뿐 아니라 매개변수 타입 등을 포함한 내부 이름(Name Mangling)을 만들어 서로 다른 함수로 구분한다. 이를 네임 맹글링(Name Mangling) 이라고 한다.

    이러한 네임 맹글링을 이용하여 함수의 이름 자체는 같으나 매개 변수의 정보에 따라 다른 함수를 불러오는 함수 오버로드가 가능하다.

    ex(1)-자료형에 따라 다른 cout을 출력하는 예제

    #include <iostream>
    using namespace std;

    void print(int a) {
        cout << "정수 출력: " << a << endl;
    }

    void print(double a) {
        cout << "실수 출력: " << a << endl;
    }

    int main() {
        print(10);     // 정수 출력
        print(3.14);   // 실수 출력
        return 0;
    }

    ex(2)-매개 변수의 갯수가 다른 연산을 진행하는 예제

    #include <iostream>
    using namespace std;

    int add(int a, int b) {
        return a + b;
    }

    int add(int a, int b, int c) {
        return a + b + c;
    }

    int main() {
        cout << add(1, 2) << endl;        
        cout << add(1, 2, 3) << endl;     
        return 0;
    }

    하지만 이러한 함수 오버로딩 또한 조건에 구애 받지 아니하고 사용할 수 있는 것은 아니다.

    ex(1)은 매개변수의 타입(자료형)이 확실히 다른 상황이고,
    ex(2)는 매개변수의 개수가 확실하게 다르기 때문에 사용할 수 있다.

    그렇다면 안되는 상황은 무엇이 있을까?


    ex(3)-형 변환이 두 함수 모두에 가능한 예제

    #include <iostream>
    using namespace std;

    void print(double a) {
        cout << "double: " << a << endl;
    }

    void print(long a) {
        cout << "long: " << a << endl;
    }

    int main() {
        print(10);
        return 0;
    }

    ex(3)은 10이라는 값이 double로도 long으로도 표준 변환(Standard Conversion)을 할 수 있으며,
    이는 프로그램에서 어느 쪽으로 변환하는 것이 올바른지 판단할 수 없기에 에러가 발생한다.


    ex(4)- 기본 값 매개변수가 존재하여 구분이 안되는 예제

    #include <iostream>
    using namespace std;

    void display(int a, int b = 5) {
        cout << a << ", " << b << endl;
    }

    void display(int a) {
        cout << a << endl;
    }

    int main() {
        display(10);
        return 0;
    }

    ex(4)는 기본값이 지정된 매개변수가 존재하며,
    이로 인해 매개변수를 1개만 입력하여도 작동하게 된다.

    따라서 두 함수 중 어느 함수를 호출한 것인지 구분할 수 없게 되며
    이로 인하여 컴파일 에러가 발생하게 된다.


    ex(5)-매개변수 포인터와 매개변수 배열의 구분이 불가능한 경우

    #include <iostream>
    using namespace std;

    void print(int* arr) {
        cout << "포인터 호출됨" << endl;
    }

    void print(int arr[]) {
        cout << "배열 호출됨" << endl;
    }

    int main() {
        int data[3] = {1, 2, 3};
        print(data);
        return 0;
    }

    ex(5)는 특수한 사항으로 함수의 매개변수에서는 배열 타입이 자동으로 포인터 타입으로 변환(decay)된다. 따라서 해당 상황은 함수 재정의에 해당하기에 컴파일에러가 발생한다.


    ex(6)- 반환 타입만 서로 다른 경우

    #include <iostream>
    using namespace std;

    // int getValue() {
    //     return 10;
    // }

    // double getValue() {
    //     return 3.14;
    // }

    int main() {
        cout << getValue();
        return 0;
    }
    
    ex(6)은 애매할 수 밖에 없다.
    네임 맹글링부터 기준점은 "매개변수"이니 반환 타입으로는 함수 오버로드가 불가능하다.

    따라서 함수 오버로드를 사용할 때에는 자동 형변환, 오버로드의 대상이 되는 함수의 기본 값 설정에 유의가 필요하며, 언제나 명확한 기준을 두고 사용하는 것이 좋아 보인다.

    그 좋은 기준 및 순서는 아래의 예시 코드들을 보며 확실하게 이해할 수 있었다.

    ex(7)-정확한 자료형 일치 우선
    #include <iostream>
    using namespace std;

    void print(int a) {
        cout << "정확한 타입(int) 호출됨" << endl;
    }

    void print(double a) {
        cout << "double 타입 호출됨" << endl;
    }

    int main() {
        print(10); // 정확한 int 타입 일치
        // 극단적인 예시로는 print((int)'A')같은 느낌으로 쓸 수 있다.
        return 0;
    }
    이것 처럼 명확하게 자료의 형변환을 한다면
    오버로드를 사용하는데 유용할 것으로 보인다.
    
    ex(8) - 손실 감소의 승격 (Promotion)

    char, short, bool → int
    float → double

    함수 오버로드에서는 가능한 한 값의 손실이 없는 승격(Promotion)이 우선적으로 선택된다.
    반대로 float → int처럼 값이 손실될 가능성이 있는 변환(Conversion)은 Promotion보다 우선순위가 낮다.

    추가 내용 (리마인드)

    자료형 변환과 관련하여 부동소수점에 대해 다시 정리한다.

    부동소수점은 0.1과 같은 값을 2진수로 정확히 표현하지 못하기 때문에,
    0.1 + 0.2의 결과가 0.30000000000000004처럼 오차가 발생할 수 있다.

    
    ex(9)-표준 타입 우선 변환

    #include <iostream>
    #include <string>
    using namespace std;

    void print(int a) {
        cout << "int 버전 호출됨: " << a << endl;
    }

    void print(string a) {
        cout << "string 버전 호출됨: " << a << endl;
    }

    int main() {
        double num = 10.99;
        
        print(num);
        return 0;
    }
    ex(9)는 표준 타입으로 우선 변환하는 규칙으로 표준 변환(Standard Conversion)은 사용자 정의 변환(User-defined Conversion)보다 우선한다.

    추가 내용:
    string은 언어마다 다르지만 표준 자료형이 아니다.
    C++에서는 클래스이며, C언어에서는 '\0'으로 끝나는 char 배열이다.

2. 사용자 정의 형변환(operator):

    해당 내용은 함수 오버로드를 공부하며 추가로 공부한 내용이다.
    해당 내용은 사용자가 만든 클래스가 함수 오버로드의 상황에서 어떻게 작동되는지에 대한 내용이다.

    함수 오버로드의 상황에서 함수에 클래스가 들어갈 경우 컴파일러가 클래스 내부에서
    operator int()나 operator float()같은 형변환을 찾아서 함수에서 클래스의 자료형을 정해준다.

    operator int()의 기능에 대한 착오가 있었는데,
    operator int()는 객체 자체가 int가 되는 것이 아니라, int가 필요한 문맥에서 객체를 int 값으로 변환할 수 있게 해주는 사용자 정의 변환 함수였다.

    그렇기에
    operator int()이외에 operator float()을 같이 적어서 int를 매개변수로 받는 함수와 float을 매개변수로 받는 함수 둘 사이에서 함수 오버로드가 발생할 경우 ambiguous call (모호한 호출)이라는 컴파일러 에러가 생기는 것도 알았다.


3. 템플릿: 
    템플릿은 하나의 코드로 여러 타입에 대해 코드를 생성하도록 하는 기능이다.
    이는 내가 과거에 배운 파이썬에서 활용하는 자동 형변환, 옛 Java의 var, C언어의 void* 와 유사한 것으로 생각했다. 하지만 템플릿은 활용하면 함수나 클래스 자체를 타입에 따라 만들어내는 느낌의 기능으로 활용은 유사하나 기능은 전혀 다르다는 것을 알았다.

    템플릿 사용 시: void add(const T& element)형태로 사용하는 것을 보았다.
    이를 보고 const에서 레퍼런스의 안정적인 사용법을 보았다. 하지만 뒤이어 이럴 것이라면 굳이 레퍼런스를 써야하는가. 하는 의문이 들어 조사해 보았다.

    해당 내용을 조사해본 결과 템플릿 활용의 주의점을 알게되며 더더욱 필요한 한줄임을 알게 되었다.
    이제부터 이에 대해 설명하겠다.

    템플릿은 자유롭게 자료형을 가진다. 이는 int같은 작은 자료형 뿐만 아니라 클래스 또한 될 수 있다는 것으로 int같은 자료형이라면 int element 같은 상황에서는 복사하는 것이 좋다. 하지만 용량이 큰 클래스의 경우 T element를 복사할 경우 프로그램 자체의 메모리 활용이 늘어나게 되며 이는 바람직하지 못하다. 그렇기에 T& element 를 활용하며 복사 없이 값을 활용하고 const를 붙여 굳이 매개변수에 넣은 값을 수정할 필요가 없다면, 매개변수를 const로 지켜 활용하는 매우 훌륭한 방법이었다.  

4. vector 기초: 
    vector는 연속된 메모리를 사용하므로 앞에 데이터를 삽입하면 뒤의 요소를 모두 이동해야 한다. 또한 vector는 어떤 타입이든 저장할 수 있으므로, 요소가 큰 객체일 경우 이동(또는 복사) 비용이 커져 성능에 영향을 줄 수 있다. 그래서 vector는 뒤에서 추가·삭제(push_back, pop_back)하는 방식이 가장 효율적이다.

    함수 오버로딩은 “컴파일러가 호출 대상을 명확히 고를 수 있어야 한다”가 핵심이다.
