    아래 코드는 Step5와 Step6을 구현하며 설계한 핵심 흐름이다.   


    void ExtractDropItem( const Monster& monster, string& itemName, int& price){

        itemName = monster.GetDropItemName();
        price = monster.GetDropItemPrice();

    }

    
    Item BuildItem( const string name, const int price )
    {

        Item item_data;

        item_data.name = name;
        item_data.price = price;

        return item_data;

    }

    
    void InsertInventory(const Item& item, vector<Item>& Inventory){

        Inventory.push_back(item);

    }


    //전투 진행 및 처치 시 아이템 등록
    ExtractDropItem( *monster, dropName, dropPrice );
    Item item = BuildItem( dropName, dropPrice );
    InsertInventory( item, Inventory );


    처음에는 Battle 함수가 Item을 직접 생성하도록 생각했지만,
    역할을 분리하는 것이 유지보수와 재사용성 측면에서 더 적합하다고 판단하여 설계를 수정하였다.

    따라서 작성자는 ExtractDropItem, BuildItem, InsertInventory로 역할을 분리하여 단일 책임 원칙(SRP)을 적용하였다. 또한 Monster의 내부 데이터는 Getter를 통해 접근하여 캡슐화를 유지했고, 필요한 데이터만 전달하는 방식으로 객체 간 결합도를 낮추는 구조를 연습하였다. 마지막으로 const 참조를 사용하여 불필요한 복사를 줄이고 함수의 의도를 명확하게 표현하는 습관을 익혔다.
